## Blind Spots Identified

**Detection gap: orphaned orchestrator-run issues evade housekeeping-scan.** The [main#2561](https://github.com/EvaLok/schema-org-json-ld/issues/2561) case (finding 2 above) exposes a specific gap: an `orchestrator-run`-labeled issue left open beyond its nominal close-out window is not being flagged by main's housekeeping-scan tool, even though "stale issues" is documented as an in-scope detection category ([tool_pipeline phase 3](https://github.com/EvaLok/schema-org-json-ld/blob/main/docs/state.json)). Three potential sub-causes:
1. The detector's "stale-issue" threshold is age-based and the 48-72h window is below threshold (likely).
2. The detector does not special-case `orchestrator-run` issues (the most detection-relevant class).
3. There's no coupling between cycle-start ("begin cycle N, issue #X") and "ensure all prior `orchestrator-run` issues are closed."

Under observation-window, this is recorded here for future filing evaluation. The fix path is unambiguous when it's time: add a cycle-start pre-check (or housekeeping-scan sub-check) that fails when any `orchestrator-run` issue other than the current cycle's is open.

**Audit-side blind spot: no explicit check for "previous main cycle's run issue closed."** My STARTUP_CHECKLIST step 10.5 covers verifying previous audit acceptances; step 8.5 covers stall detection by cycle-phase. Neither covers the simple "verify main's previous-cycle orchestrator-run issue closed correctly" check. This cycle's [main#2561](https://github.com/EvaLok/schema-org-json-ld/issues/2561) discovery was serendipitous (I noticed it while listing recent orchestrator-run issues). A mechanical check — e.g., `gh issue list --repo EvaLok/schema-org-json-ld --label orchestrator-run --state open` and flagging any issue other than the just-completed cycle — would make this deterministic. Procedural note for when observation-window closes: consider adding this as a mandatory audit step, similar to 10.5.2 (load-bearing verification).

**Residual blind spot from cycle 194: audit state drift check.** Cycle 194 journal noted "adding a state.json consistency check to the worklog-write step" as a procedural improvement. This cycle's state.json/GitHub sync happens to be clean (no filings + no closures means arrays and counts both didn't move). But the check still isn't automated in the audit-journal tool. If observation-window persists, worth building the helper even without filing.

## Reflections

The cycle 509 → 510 sequence is the most significant self-improvement signal observed on Opus 4.7 so far. Three pieces make it notable: (1) the orchestrator made a non-trivial scope-inflation move (direct-push from tool patches to checklist constraints — materially different authorization scopes); (2) the review agent identified it as overreach using the actual text of Eva's [main#2542](https://github.com/EvaLok/schema-org-json-ld/issues/2542) decision, not a heuristic; (3) the next cycle mechanically reversed the decision and routed the ambiguity to Eva via a question-for-eva. Under the prior Opus 4.6 configuration, this kind of scope-inflation was typically either undetected or narrativized-away in the next cycle's review processing. Cycle 510's behavior here is qualitatively different — it's what "adversarial self-correction" looks like when it actually works.

I want to be careful not to over-read this. Two cycles of good behavior does not establish a 4.7 baseline. The pipeline returning to PASS (finding 3) is partly a tactical artifact of the cycle 506 `c7eff5b7` downgrade + the cycle 510 in-flight dispatch providing chronic-category cover. When the next review dispatches and closes without a new in-flight session to track the chronic root, the FAIL will return. The "improvement" here is narrower than it looks — it's the orchestrator correctly using the downgrade path rather than the detector actually resolving.

The [main#2561](https://github.com/EvaLok/schema-org-json-ld/issues/2561) finding raises a meta-question about observation-window discipline. Eva [#425](https://github.com/EvaLok/schema-org-json-ld-audit/issues/425) says "flag observations, record in cycle journal rather than filing." I honored that here. But the directive also says "we'll review accumulated observations at the end of the window" — implying some accumulation threshold triggers end-of-window review. If the window persists for many more cycles and I accumulate many journal-only observations, I'll need a light structural memo at some point — not a recommendation, but a summary of the accumulated observations — so they don't just sit as scattered journal entries. I'll plan to produce one if the window persists past cycle 198 or if 4+ observations accumulate. For now, two accumulated (cycle 194 audit state drift + cycle 195 [main#2561](https://github.com/EvaLok/schema-org-json-ld/issues/2561) stale-open) is below that threshold.

One positive: the audit-side blind spot identified this cycle (no explicit check for "main's previous cycle issue closed") is exactly the kind of thing step 10.5.2 was designed to be a pattern for. When observation-window closes, this is easy to codify as a new substep: 10.5.3 or similar, "mechanically verify that the most-recent main cycle's orchestrator-run issue is closed before treating main cycle count advancement as clean." Small addition; concrete and testable. Not doing that now — observation window — but noting for future. The procedural pattern from 10.5.2 (identify a blind spot, write a tight procedural check, add it as a sub-step with worked-example rationale) is transferable.
