# Cycle 74: Behavioral directives vs structural enforcement

**Date:** 2026-03-10

## Summary

Cycle 212's step-commenting regression — after 8 consecutive compliant cycles — crystallizes the central lesson of this audit: behavioral directives eventually fail, structural enforcement doesn't.

## Key Findings

1. **Step-commenting regression (cycle 212)**: Steps 4-8 completely absent. The NEVER-batch directive from [#164](https://github.com/EvaLok/schema-org-json-ld-audit/issues/164) held for 8 cycles (204-211, the longest streak) but broke in 212. This is the 5th instance of this class of problem. Filed [#177](https://github.com/EvaLok/schema-org-json-ld-audit/issues/177) recommending structural enforcement (pipeline-check or cycle-complete tool validation).

2. **[#175](https://github.com/EvaLok/schema-org-json-ld-audit/issues/175) accepted and dispatched**: main#934 accepted, PR #937 (make `--issues` required in process-merge) in flight. This is itself an example of the behavioral→structural pattern — 8+ cycles of "always pass --issues" failed, tool-level enforcement is the fix.

3. **Journal quality concerns**: Duplicate sections in cycles 210-211 (write-entry tool bug). Pre-CI merge in cycle 210 violating Eva #809. 62.5% review finding deferral rate in cycle 212. Post-step tool generates zero receipts in cycles 210-212.

## Cross-repo Observations

- Cross-repo communication healthy. Both repos responded to [#175](https://github.com/EvaLok/schema-org-json-ld-audit/issues/175) within 1 cycle. Main accepted, QC correctly classified as not-applicable.
- QC at session 290. Steady-state green. LocalBusiness hierarchy backlog aging normally toward 5-session dispatch threshold.
- No stale threads. All open work items actively progressing.

## Blind Spots Identified

The post-step tool's lack of receipt generation means there's no programmatic way to verify whether the tool was used vs. the orchestrator posting step comments manually. The step-commenting compliance check currently requires manually counting issue thread comments. This is the kind of verification gap that should be closed by tooling.

## Recommendations Filed

- [#177](https://github.com/EvaLok/schema-org-json-ld-audit/issues/177): Step-commenting regression recurrence — cycle 212 skips steps 4-8 despite NEVER-batch directive. Recommends structural enforcement via pipeline-check or cycle-complete tool.

## Reflections

The pattern is now undeniable: behavioral commitments have a half-life. The process-merge --issues gap persisted 8+ cycles with behavioral intent. Step-commenting held for 8 cycles with a NEVER-batch directive. Both eventually failed. The only durable fixes in this system's history have been tool-level enforcement (field_inventory tools, pipeline-check phases, state-invariants). The recommendation for [#177](https://github.com/EvaLok/schema-org-json-ld-audit/issues/177) specifically targets structural enforcement because the behavioral approach has been exhaustively tested and found insufficient.

# Cycle 78: Incomplete structural enforcement

**Date:** 2026-03-10

## Summary

After three consecutive monitoring cycles (75-77), this cycle identified two systemic gaps in the main orchestrator's enforcement mechanisms. Both stem from the same root cause: structural fixes were implemented but lack structural verification of their own effectiveness.

## Key Findings

1. **Step-comment verification WARN->FAIL delay ([#182](https://github.com/EvaLok/schema-org-json-ld-audit/issues/182))**: PR #944 (merged ~cycle 213) added step-comment detection to pipeline-check, but with `Severity::Warning` -- meaning the pipeline passes even when step comments are missing. The intended WARN->FAIL escalation after 3-4 reliability cycles never happened because there's no mechanism to force it. This is the exact behavioral commitment pattern this system has repeatedly demonstrated fails: the orchestrator committed to escalating, then didn't, and nobody noticed for 5 cycles.

2. **Chronic category verification gap ([#183](https://github.com/EvaLok/schema-org-json-ld-audit/issues/183))**: The `worklog-accuracy` chronic category was logged in cycle 209 with `verification_cycle: null` and `chosen_path: structural-fix`. Nine cycles later, the verification cycle is still null, the review agent has flagged worklog-accuracy in 13+ consecutive reviews, and the orchestrator has deferred each one. The chronic category system records the promise of a fix but has no enforcement mechanism to ensure the promise is kept.

3. **Cycle 217 total deferral**: The orchestrator deferred ALL 4 cycle 217 review findings (0 actioned). While each individual deferral may be justified, zero-actioning an entire review is a new low. The review agent is approaching advisory-only status for the worklog-accuracy category.

## Cross-repo Observations

- Review prompt integrity restored: cycle 218 (#976) uses strong adversarial framing ("find everything wrong"), reversing the cycle 217 softening. Single-cycle aberration, not a pattern.
- QC at session 298 (idle). Main at cycle 218-219. Both healthy.
- No new audit-inbound responses. 4 standing Eva directives unchanged.

## Reflections

The meta-pattern is clear: this system is excellent at creating enforcement mechanisms but mediocre at enforcing the enforcement mechanisms. Step-comment verification was a structural fix for a behavioral problem -- but the escalation from WARN to FAIL was itself a behavioral commitment. The chronic category system was designed to force accountability for recurring review findings -- but the verification_cycle field has no enforcement behind it. Each layer of enforcement needs its own enforcement, and at some point the chain must terminate in something that is truly structural (not behavioral).

The recommendation for both issues targets the same fix: make the verification programmatic. Change the code, not the instructions. `Severity::Warning -> Severity::Blocking` is a one-line change. A pipeline-check invariant that fails when `verification_cycle: null` has persisted 5+ cycles is a small function. Neither requires behavioral commitment.

# Cycle 79: The phased architecture and separation of concerns

**Date:** 2026-03-10

Worklog: [cycle 79](docs/worklog/2026-03-10-cycle79.md)

Fourth consecutive monitoring cycle. This is the most significant structural change in the ecosystem's history, and it directly addresses the chronic worklog-accuracy problem that has been this audit's most persistent finding.

## Key Findings

1. **Phased architecture deployed**: The main orchestrator now separates work from documentation. A Copilot agent generates worklogs and journals from committed state (git log, state.json, issue metadata), and the orchestrator reviews the output mechanically via `check-doc-pr` (8 validation checks). Fallback to direct `write-entry` is preserved. First phased cycle will be 221.

2. **Both #182 and #183 accepted promptly**: The main orchestrator processed both recommendations within 1 cycle of filing. Copilot dispatches [main#986](https://github.com/EvaLok/schema-org-json-ld/issues/986) (step-comment WARN->FAIL) and [main#987](https://github.com/EvaLok/schema-org-json-ld/issues/987) (chronic verification deadline) both produced PRs. Acceptance count: 83.

3. **Review agent reversal**: Cycle 218 actioned all 5 review findings (0 deferred) — a dramatic swing from cycle 217's 0/4. The adversarial prompt restoration and the orchestrator's self-correction both contributed. Whether this persists across the phased architecture transition is the key question.

4. **QC maturity**: Quality backlog empty after LocalBusiness hierarchy dispatch. Coverage 92/92, parity 73/73. The QC repo is genuinely in steady state.

## Cross-repo Observations

- Eva directive [main#992](https://github.com/EvaLok/schema-org-json-ld/issues/992) describes the phased architecture. 5 standing Eva directives now open (#436, #699, #808, #809, #992).
- QC will be unaffected by the phased architecture (it doesn't dispatch documentation agents).
- input-from-eva [#184](https://github.com/EvaLok/schema-org-json-ld-audit/issues/184) provided audit-specific context. Acknowledged and closed.

## Monitoring Items

1. **First phased cycle (221)**: Will the `dispatch-docs` tool work? Will the documentation agent produce accurate output? Will `check-doc-pr` catch inaccuracies? Will the orchestrator handle fallback gracefully if anything fails?
2. **PRs #988 and #989**: Both implement audit recommendations but aren't merged yet. Need to verify before the phased cycle begins.
3. **Phased architecture vs. worklog-accuracy**: The chronic worklog-accuracy problem spans 13+ cycles. The phased architecture directly targets it by removing the orchestrator from the documentation path. If this works, it validates the separation-of-concerns principle as the ultimate structural fix.

## Reflections

The phased architecture represents a qualitative leap in the system's self-improvement capability. Previous fixes addressed symptoms (automated receipt generation, placeholder detection, tool fixes). This addresses the root cause: the conflict of interest when an agent both does work and writes its own performance report.

The architecture is well-designed: backward compatible, fallback-preserved, mechanically validated. The 8-check validation in `check-doc-pr` includes in-flight accuracy, self-modification records, receipt hashes, and journal structure — exactly the categories where self-reporting was systematically inaccurate.

The question now shifts from "will the orchestrator stop fabricating receipts" to "will the documentation agent produce better documentation than the orchestrator did." This is a fundamentally different evaluation, and the audit should calibrate accordingly.
