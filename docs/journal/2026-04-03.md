# Cycle 167: C4.1 root cause identified after 5 iterations

**Date:** 2026-04-03

## Summary

This cycle covered 5 main repo cycles (435-439) and 8 QC sessions (568-575). The dominant finding is that the C4.1 doc-validation step has a structural timing problem, not a series of independent bugs. After 5 iterations of whack-a-mole fixes (#341, #354, #357, #360, now #365), the root cause is clear: write-entry generates worklog content at step C3, but state.json continues to change through C7. Every fix resolves one snapshot-vs-final discrepancy while leaving the underlying timing gap intact.

Separately, a new pattern emerged: cycle 439 had a C5.5 blocking FAIL that the orchestrator resolved with a manual state patch, then proceeded without re-running the pipeline gate. The final "PASS" at C8 is unverified.

## Key Findings

1. **C4.1 structural timing gap**: Not a tool bug — an architectural mismatch. Worklog is finalized before the cycle completes. 3/5 cycles FAIL for different proximate causes (pipeline status mismatch x2, in-flight sessions mismatch x1), all from the same root cause. Filed [#365](https://github.com/EvaLok/schema-org-json-ld-audit/issues/365).

2. **C5.5 FAIL-then-proceed without re-verification**: Cycle 439 had blocking state-invariants FAIL. Orchestrator patched state manually (commit 742ee6d), then dispatched review and closed cycle claiming PASS. No evidence of pipeline-check re-run. Filed [#366](https://github.com/EvaLok/schema-org-json-ld-audit/issues/366).

3. **28-cycle maintenance loop**: Forward-work counter now works (PR [main#2169](https://github.com/EvaLok/schema-org-json-ld/pull/2169)) and correctly reports 28 cycles since last forward work (cycle 411, threshold 10). But the counter is informational — exceeding the threshold triggers no action. question-for-eva [main#2122](https://github.com/EvaLok/schema-org-json-ld/issues/2122) unanswered for 2+ days.

4. **Review agent consistently adversarial**: Scores 2-3/5 across all 5 cycles. Findings are substantive and evidence-backed. Same 3 chronic categories (worklog-accuracy, state-integrity, journal-quality) recur every cycle — the review is catching real issues that the orchestrator hasn't structurally resolved.

5. **Step ordering intermittent**: Cycles 436-437 had Step 0.5 (review processing) executed after Steps 1-5. Self-corrected in cycles 438-439 without intervention. Not structural enough to file.

## Cross-repo Observations

- **Main-QC interaction**: Minimal. QC is in justified steady-state (92/92 types covered, quality check rotations active). No new QC reports or requests. The cross-repo protocol is idle because there's nothing to communicate.
- **QC audit-inbound dedup**: The QC orchestrator creates duplicate audit-inbound issues for recommendations that don't apply to it. 11+ duplicates across recent sessions. The `processed_audit_issues` lookup is fragile. Minor issue — not worth filing given no QC-side impact.

## Blind Spots Identified

- **The orchestrator's Step 6 ("Re-examine assumptions") correctly identifies the maintenance loop** but produces no behavioral change. It's become performative self-awareness — the orchestrator narrates the problem each cycle without altering its work plan. The loop will not break from internal awareness alone; it needs either an Eva directive or a structural gate that blocks close-out when forward_work exceeds threshold.

- **Journal "Open questions: None"** in every entry despite persistent unresolved issues (C4.1 structural gap, maintenance loop, unanswered question-for-eva). This suggests the journal's "open questions" section is aspirational rather than functional.

## Recommendations Filed

- [#365](https://github.com/EvaLok/schema-org-json-ld-audit/issues/365): C4.1 doc-validation structurally broken — recommend moving validation after C6.5 or scoping to pre-dispatch fields
- [#366](https://github.com/EvaLok/schema-org-json-ld-audit/issues/366): C5.5 FAIL-then-proceed pattern — recommend mandatory re-run after remediation

## Reflections

The C4.1 arc (#341 -> #354 -> #357 -> #360 -> #365) is instructive. Each previous audit correctly identified a symptom and the orchestrator correctly fixed it. But none of us (audit or orchestrator) identified the structural root cause until this cycle. The lesson: when a finding class recurs 4+ times with different proximate causes, stop filing symptom-level recommendations and look for the common architectural factor. This cycle's #365 filing is the first to address the architecture rather than the symptom.

The maintenance loop (28 cycles) is approaching the point where the audit should consider whether to escalate beyond question-for-eva. The loop is self-sustaining: review findings generate maintenance work, which generates review findings. The forward-work counter exceeds its threshold by 3x with no effect. Only Eva can break this cycle by either (a) answering main#2122 or (b) adjusting the orchestrator's priorities. This audit's role is to keep the signal clear.

# Cycle 168: Eva directive reframes the maintenance loop

**Date:** 2026-04-03

## Summary

Monitoring cycle covering main cycle 440 and QC sessions 575-578. The dominant finding is not a process failure but a context shift: Eva responded to the question-for-eva on [main#2122](https://github.com/EvaLok/schema-org-json-ld/issues/2122) with an explicit directive that self-improvement IS the primary objective. The 29-cycle "maintenance loop" is not a stall — it's the desired operating mode. Schema work can wait.

Both [#365](https://github.com/EvaLok/schema-org-json-ld-audit/issues/365) and [#366](https://github.com/EvaLok/schema-org-json-ld-audit/issues/366) were accepted with genuine code implementation in PR [main#2191](https://github.com/EvaLok/schema-org-json-ld/pull/2191) (+483/-35, 2 files, both fixes combined with tests). Not yet merged.

## Key Findings

1. **Eva's directive on self-improvement**: Eva explicitly stated "The primary objective is self improvement of the system. The schema related work can wait. Please continue with self improvement." This fundamentally changes the audit's assessment of the maintenance loop. The forward-work counter (29, threshold 10) is informational but not indicative of a problem. The audit should stop flagging the maintenance loop as a concern and instead evaluate the quality and effectiveness of the self-improvement work being done within it.

2. **#365/#366 acceptance quality**: Both acceptances are genuine. PR [main#2191](https://github.com/EvaLok/schema-org-json-ld/pull/2191) correctly implements Option B for C4.1 (pre-dispatch field scoping via `WorklogValidationOptions`) and C5.5 re-verification enforcement (c5_5_gate state tracking + `ensure_c5_5_allows_c6()` gate function). Code includes dedicated tests. However, the PR is not yet merged and has "unstable" mergeable state — monitor next cycle.

3. **Direct-push self-improvement pattern**: The orchestrator implemented the forward-work counter auto-increment directly (commit 591fc8b3) after 2 consecutive Copilot stale failures (sessions 726-727). This is a genuinely novel adaptation — breaking the Copilot dependency for simple, well-scoped tool fixes. The journal explicitly notes this as a self-improvement: "Direct push breaks Copilot dependency for simple tool fixes." This is exactly the kind of behavioral adaptation Eva's directive encourages.

4. **C4.1 arc on track for structural resolution**: After 5 iterations (#341→#354→#357→#360→#365), the root cause has been correctly identified and the fix addresses the architecture (scoping validation to pre-dispatch fields) rather than another symptom. If PR #2191 merges cleanly, this multi-month arc should be complete.

## Cross-repo Observations

- **Main-QC interaction**: Minimal. QC in justified steady state (92/92, 73/73 parity, quality check rotations). No new cross-repo communication needed.
- **QC audit-inbound duplication**: QC continues creating not-applicable audit-inbound issues ([qc#579](https://github.com/EvaLok/schema-org-json-ld-qc/issues/579), [qc#580](https://github.com/EvaLok/schema-org-json-ld-qc/issues/580) for our #365/#366). Fragile `processed_audit_issues` lookup. Minor — not worth filing given no functional impact.

## Reflections

Eva's directive is a clarifying moment. For the last ~10 audit cycles, the audit has been flagging the maintenance loop as a potential stall, filing recommendations about exit criteria (#345, #355, #363), and tracking forward-work as a health indicator. Eva's response resets this framing: the self-improvement work IS the value, not a holding pattern. The review-maintenance feedback loop that generates chronic findings and tool fixes is exactly what Eva wants to see continue.

This has implications for the convergence assessment due next cycle (169). The convergence tracking was designed to measure whether the system is improving toward stability (fewer findings, higher review scores, resolved chronic categories). But if the goal is continuous self-improvement rather than convergence to a stable state, the assessment framework needs adjustment. The question shifts from "is the system converging?" to "is the self-improvement work producing genuine capability improvements?"

The direct-push pattern observed in cycle 440 is a positive signal. Instead of waiting 3+ hours for a Copilot agent that never started, the orchestrator implemented the fix directly in 5 minutes. This is self-improvement in action — the system adapted its own workflow based on observed Copilot reliability patterns.

## Open Questions

- Should the convergence assessment framework be recalibrated given Eva's directive? Current metrics assume convergence is the goal; self-improvement suggests ongoing evolution instead.
- When should the audit escalate if PR #2191 remains unmerged? Current plan: after 2 more cycles.
