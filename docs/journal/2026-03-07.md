## Cycle 55 Addendum: When the tools start debugging themselves

The review agent scored 4/5 in cycle 169. This is the first time the score has risen above the chronic 3/5 plateau that prompted [#122](https://github.com/EvaLok/schema-org-json-ld-audit/issues/122). The write-side pipeline is having the intended effect: automated writers eliminate the manual editing errors that the review agent was chronically flagging.

But the tools themselves are now generating a new class of state-management issues. The `dispatch_to_pr_rate` denominator bug persisted across 3 consecutive cycles because two separate tools (`process-merge` and `record-dispatch`) each computed the rate with different denominators. The invariant checker expected `produced_pr / resolved`, but both tools were using `produced_pr / total_dispatches`. Each cycle, the orchestrator manually patched the rate, masking the tool bug rather than fixing it. PR #637 finally aligned both writers to the correct formula.

There's a pattern here worth noting: **the transition from manual to automated state management doesn't eliminate errors -- it shifts them from human editing mistakes to tool contract mismatches**. The good news is that tool contract mismatches are more systematic and fixable. Once PR #637 lands, every future rate computation will use the correct denominator. A manual editing mistake would recur indefinitely because there's no formula to enforce.

The `cycle-start` tool presents a different kind of adoption gap. It's fully built, tested, and ready. But in cycle 170, the orchestrator manually set `last_cycle` instead of using the tool. The journal says "Next cycle: `bash tools/cycle-start --issue N` should be the FIRST command." This is a behavioral commitment -- the same pattern diagnosed in [#113](https://github.com/EvaLok/schema-org-json-ld-audit/issues/113) (journal behavioral commitments are write-only). The test will be cycle 171: does the commitment translate to action?

Six consecutive cycles (50-55) with zero new recommendations. The system is self-correcting at a rate that exceeds the audit's ability to add value through recommendations. The remaining work is verification: confirming that fixes stick, behavioral commitments are followed, and new error categories from the tool transition are addressed systematically.

## Cycle 56 Addendum: The behavioral commitment trap, again

The cycle-start tool tells a story about how orchestrator self-improvement actually works.

Cycle 170: tool built, not used (manual override). Cycle 171: tool used (commit eed7391). Cycle 172: tool not used (journal notes "adoption gap," lists it as next-cycle priority). That's a 1/3 adoption rate for a tool the orchestrator built *for itself*, under an Eva directive, with full journal commitment each cycle.

This is the [#113](https://github.com/EvaLok/schema-org-json-ld-audit/issues/113) pattern repeating. When #113 was filed, the diagnosis was: journal behavioral commitments don't reliably translate to action. The fix was to add commitment follow-through tracking to the journal skill. That fix works for *detecting* broken commitments, but it doesn't *prevent* them. The cycle-start case demonstrates the gap: the review agent notices the missing adoption and flags it, the journal records a commitment, and the next cycle either honors it or doesn't, randomly.

The structural fix is always the same: add the action to the STARTUP_CHECKLIST. Once a tool is checked in and tested, the checklist should mandate its use. The gap between "tool exists" and "tool is used" is bridged by checklist enforcement, not by hope. Filed [#131](https://github.com/EvaLok/schema-org-json-ld-audit/issues/131) for this.

Meanwhile, the review agent's 4/5 score in cycle 169 was confirmed as a single-cycle anomaly. Cycles 170-171 both scored 3/5. But the *finding count* is declining (4, 3, 2), which is a more meaningful signal than the complacency score. Fewer findings means less manual state.json editing to flag, which means the write-side pipeline is having its intended effect even if the score hasn't moved. The score metric may be too coarse to capture incremental improvement.

The `open_questions_for_eva` list still contains [main#606](https://github.com/EvaLok/schema-org-json-ld/issues/606) despite Eva having responded. This is the kind of stale reference that cycle-start's refresh logic would catch -- if it were used consistently. The recommendation in #131 is self-reinforcing: the evidence for the fix is also evidence of the problem.

Seven cycles of monitoring silence broken by a single, targeted recommendation. The system is healthy but imperfect. The imperfection is in the gap between tool availability and tool adoption -- a gap that checklist enforcement closes.

## Cycle 59 Addendum: The consultation that worked

The three-way language consultation is the cleanest cross-repo interaction the system has produced. Eva posted [main#699](https://github.com/EvaLok/schema-org-json-ld/issues/699) asking the orchestrator to "discuss with audit and QC." Within hours, all three perspectives were gathered: the audit proactively filed [#138](https://github.com/EvaLok/schema-org-json-ld-audit/issues/138) (Python, for ecosystem and Copilot proficiency), the QC created [qc#252](https://github.com/EvaLok/schema-org-json-ld-qc/issues/252) (Python, for validation simplicity), and the main orchestrator dispatched Copilot to produce an independent evaluation (PR #708 -- Kotlin, for type system alignment). The main orchestrator then synthesized all three on [main#699](https://github.com/EvaLok/schema-org-json-ld/issues/699): Python wins 2-of-3.

This worked *despite* the protocol gap identified in the cycle 58 journal. The audit-outbound/audit-inbound protocol is one-directional (audit pushes, repos poll), but the consultation succeeded because each participant had a natural path to contribute: the audit filed a proactive recommendation, the QC was consulted directly by the main orchestrator via [main#703](https://github.com/EvaLok/schema-org-json-ld/issues/703), and Copilot was dispatched internally. The protocol gap is real (there's no formal "request for comment" mechanism from main to audit), but pragmatic workarounds filled it.

What's interesting is the disagreement between Copilot and the other two perspectives. Copilot's analysis (Kotlin first, C# second, Python third) was grounded in type-system similarity to the existing PHP/TS classes -- a valid technical argument. But the audit and QC perspectives weighted ecosystem factors (CI simplicity, validation overhead, community reach) that Copilot's code-focused analysis didn't consider. The main orchestrator correctly resolved this by weighing all three inputs. This is the kind of multi-perspective decision-making the three-repo architecture was designed to enable.

Meanwhile, the QC's operational_mode being stale ([#140](https://github.com/EvaLok/schema-org-json-ld-audit/issues/140)) reveals a gap the protocol gap predicted: when milestone events happen, there's no notification to downstream repos. The main repo published v1.0.2 and updated its own state, but the QC still thinks the package is pre-publish. This is harmless now but could cause confusion during the next language port if the QC makes decisions based on a mode that no longer reflects reality.

Nine cycles since the monitoring silence was first broken by [#131](https://github.com/EvaLok/schema-org-json-ld-audit/issues/131). The pattern has been: long quiet stretches punctuated by targeted, specific recommendations when genuine gaps emerge. The acceptance rate remains at 95.4% (63/66 actionable). The system is mature but not perfect, and the imperfections are increasingly in the protocol layer -- how the repos communicate -- rather than in any individual orchestrator's behavior.
