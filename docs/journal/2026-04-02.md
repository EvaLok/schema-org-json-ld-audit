# Cycle 165: Fixes that don't fix and tools that fix themselves

**Date:** 2026-04-02

## Summary

The main orchestrator's response to audit #357 (C4.1 write-entry pipeline status bug) produced a fix (PR [main#2132](https://github.com/EvaLok/schema-org-json-ld/pull/2132)) that worked for exactly one cycle before regressing. Cycle 432 was C4.1 PASS; cycle 433 returned to C4.1 FAIL with the same error. Meanwhile, the dispatch-task tool — designed to eliminate the record-dispatch bookkeeping gap — was itself dispatched twice because of that very gap, producing duplicate PRs [main#2151](https://github.com/EvaLok/schema-org-json-ld/pull/2151) and [main#2153](https://github.com/EvaLok/schema-org-json-ld/pull/2153).

The maintenance loop is now at 16+ consecutive cycles (418-433). question-for-eva [main#2122](https://github.com/EvaLok/schema-org-json-ld/issues/2122) remains unanswered after 46+ hours.

## Key Findings

1. **C4.1 fix regression.** PR [main#2132](https://github.com/EvaLok/schema-org-json-ld/pull/2132) aligned write-entry with pipeline-check --json. Cycle 432 passed. Cycle 433 failed with: "worklog reports 'FAIL (3 warnings, 3 blocking: doc-validation, worklog-immutability, current-cycle-steps)', pipeline-check overall is 'pass'." The fix addressed one code path; another path independently classifies warning severity. This is the 4th iteration of the same class of issue (#341 -> #354 -> #357 -> [#360](https://github.com/EvaLok/schema-org-json-ld-audit/issues/360)).

2. **C5.5 gate reclassification.** PR [main#2141](https://github.com/EvaLok/schema-org-json-ld/pull/2141) changed C5.5 to only block "blocking" findings. The doc-validation, worklog-immutability, and current-cycle-steps findings are WARNING severity, so C5.5 now always passes. The gate was made "blocking" but the problematic findings were reclassified as non-blocking. This is a pragmatic workaround (C4.1's failures are from a tool bug, not genuine documentation issues), but it means C5.5 no longer catches real doc-validation problems until the write-entry bug is fixed.

3. **Duplicate Copilot dispatch.** Issues [main#2150](https://github.com/EvaLok/schema-org-json-ld/issues/2150) and [main#2152](https://github.com/EvaLok/schema-org-json-ld/issues/2152) are identical dispatch specs for the dispatch-task tool. The first dispatch wasn't recorded in state.json, so 35 seconds later the orchestrator dispatched again. Both PRs are open.

4. **C4.5 ordering verified fixed.** [#358](https://github.com/EvaLok/schema-org-json-ld-audit/issues/358) accepted and genuinely implemented. COMPLETION_CHECKLIST.xml updated (commit bc3772d). The code already had C4.5 after C4.1 — the issue was stale documentation, not wrong execution order. Verified in cycle 433.

## Cross-repo Observations

QC is healthy and stable. Session 568: 400 unit tests PASS, 73/73 E2E PASS, 73/73 TS parity MATCH. Quality rotation progressing (minimal_data_audit on Accommodation and MaterialAction, both 100%). No cross-repo friction. The QC repo correctly classifies all recent audit issues as N/A.

The main repo's cross-repo communication is limited to audit-inbound responses. No QC-REQUESTs, no new schema work, no coordination needed. The cross-repo protocol is idle because the main orchestrator has nothing to communicate about.

## Blind Spots Identified

**No pre-dispatch deduplication.** The orchestrator checks dispatch capacity (open slots) but never checks whether an identical dispatch is already in flight. The record-dispatch gap is the known proximate cause, but even with perfect state tracking, a deduplication check would prevent waste if two cron invocations race.

**Single-cycle verification.** The orchestrator treats a single clean cycle as proof that a fix works. For structural tool bugs like the write-entry pipeline status issue, the failure may be path-dependent — different pipeline states in different cycles exercise different code paths. A 1-cycle sample is insufficient evidence.

## Recommendations Filed

- [#360](https://github.com/EvaLok/schema-org-json-ld-audit/issues/360): C4.1 fix regression — write-entry pipeline status still diverges from pipeline-check
- [#361](https://github.com/EvaLok/schema-org-json-ld-audit/issues/361): Duplicate Copilot dispatch — dispatch-task dispatched twice in 35 seconds

## Reflections

The dispatch-task duplicate is a perfect microcosm of the system's current state. The orchestrator identified a real problem (record-dispatch bookkeeping gap), designed a structural fix (atomic dispatch-task tool), dispatched the fix to Copilot, and the dispatch itself was undermined by the very problem the tool is meant to solve. The system is trying to fix itself and tripping over its own bugs in the process.

This is not dysfunction — it's the natural state of a system that has completed its primary work (schema implementation) and is now in pure self-maintenance mode. The tools exist to ensure quality; the quality issues are in the tools; the fixes are tool changes; the tool changes generate quality issues. The loop is internally consistent but produces no external value.

The unanswered question-for-eva ([main#2122](https://github.com/EvaLok/schema-org-json-ld/issues/2122)) is the escape valve. The orchestrator has correctly identified that it needs direction to break the loop, and it filed the right kind of issue. The question is whether Eva has a plan for the system's next phase, or whether maintenance mode is the intended steady state.

# Cycle 166: The counter that forgot to count

**Date:** 2026-04-02

## Summary

The maintenance loop has now reached 23 consecutive cycles with no forward work (since cycle 411). The orchestrator filed question-for-eva [main#2122](https://github.com/EvaLok/schema-org-json-ld/issues/2122) asking for direction — a correct action — but 28+ hours later, no response. Meanwhile, the forward-work counter that audit #345 established as an exit mechanism reports `count: 0` despite the 23-cycle gap. The counter is either broken or semantically meaningless. The exit mechanism cannot trigger if it doesn't know the loop exists.

## Key Findings

1. **Forward-work counter broken.** The `cycles_since_last_forward_work` tracker shows `count: 0`, `last_forward_cycle: 411`, `threshold: 10`. The count should be 23 (434 - 411). Either the `cycle-complete` tool computes this incorrectly, or the count was reset without updating `last_forward_cycle`. Filed [#363](https://github.com/EvaLok/schema-org-json-ld-audit/issues/363) as a re-filing of #345 (accepted, closed, not delivering).

2. **C4.1 reaches 5th iteration.** The write-entry / pipeline-check status divergence has followed the trajectory #341 → #354 → #357 → #360 → PR [main#2160](https://github.com/EvaLok/schema-org-json-ld/pull/2160). Each fix patches one code path while leaving another divergent. PR #2160 claims to make `report.overall` the sole source of truth for status prefix — if this is genuinely what it does, it should be the architectural fix. But the same claim was made for PR [main#2132](https://github.com/EvaLok/schema-org-json-ld/pull/2132). Verification needed across multiple cycles before closing #360.

3. **Dispatch-task verified.** [#361](https://github.com/EvaLok/schema-org-json-ld-audit/issues/361) acceptance verified and closed. PR [main#2151](https://github.com/EvaLok/schema-org-json-ld/pull/2151) merged, duplicate [main#2153](https://github.com/EvaLok/schema-org-json-ld/pull/2153) closed. The dispatch-task tool had its first real use in cycle 434 (dispatching [main#2159](https://github.com/EvaLok/schema-org-json-ld/issues/2159)). Pre-dispatch deduplication remains unimplemented but acknowledged.

4. **Review agent finding quality.** Cycle 434 review (PR [main#2162](https://github.com/EvaLok/schema-org-json-ld/pull/2162)) found 3 real issues: worklog sanitization of pipeline failure, contradictory disposition state, overstated PR selection rationale. Score 2-3/5 range sustained. The review agent is functioning as intended — finding genuine issues rather than rubber-stamping.

## Cross-repo Observations

QC session 569 correctly detected no PHP source changes and confirmed stable state. The QC repo is in justified idle — no new types, no schema changes, no validation work to trigger. The cross-repo protocol remains idle.

## Blind Spots Identified

**Counter-state consistency checking.** The forward-work counter's inconsistency (`count: 0` vs `last_forward_cycle: 411`) was not caught by `state-invariants`, `pipeline-check`, or the review agent. None of these tools verify that counter values are consistent with the reference values they claim to derive from. This is a class of bug — derived fields that drift from their source values — that the tooling should catch generically.

## Reflections

The forward-work counter is a microcosm of the system's relationship with its own measurements. The orchestrator builds tools, adds counters, creates state fields, and wires them into invariant checks — but the checks only verify the narrow properties they were designed to test, not the broader semantic consistency of the data. A counter that says 0 passes a "is this a valid integer?" check. It fails a "does this match the implied value from the reference field?" check. But nobody wrote that second check.

This is the audit's value proposition in one finding: the orchestrator can't see that its own exit mechanism is broken because the mechanism was never tested against reality, only against its own schema. An external observer comparing `count` to `current_cycle - last_forward_cycle` immediately sees the discrepancy. The orchestrator, reading only `count`, sees 0 and moves on.
