## Cycle 133: Ghost cycle surfaces a new enforcement blind spot

The first audit cycle after Eva's enforcement fix (commit `2e5f160`) made Step 1.1 failures blocking. ### Finding

Main repo issue [main#1443](https://github.com/EvaLok/schema-org-json-ld/issues/1443) was triggered at 06:20 UTC but has zero comments after 2+ hours — a ghost cycle. The orchestrator session silently failed before posting any step comments or even the cycle-start comment.

This exposed a fundamental gap: **all enforcement mechanisms live inside the orchestrator session.** If the session never starts, pipeline-check never runs, step-comment verification never fires, MANDATORY_STEP_IDS has nothing to check, and the entire enforcement infrastructure is bypassed. Filed [#288](https://github.com/EvaLok/schema-org-json-ld-audit/issues/288) — ghost cycle detection. The recommendation proposes stale trigger detection in STARTUP_CHECKLIST and/or a workflow-level machine comment on issue creation.

### Implication

Eva's enforcement fix may be working perfectly — but we can't verify it because the first test cycle never ran. The counter remains at 0/50 and Eva directive [main#1442](https://github.com/EvaLok/schema-org-json-ld/issues/1442) (Phase 2a authorization) is unprocessed after 3+ hours. Cycle 298 ([main#1444](https://github.com/EvaLok/schema-org-json-ld/issues/1444)) is the next opportunity to verify.

### Observation

The pattern of infrastructure gaps found at system boundaries continues: the audit escalation path for Step 1.1 was found at the boundary between orchestrator and Eva (#284 catch-22). Now the ghost cycle gap exists at the boundary between cron workflow and orchestrator session — neither side monitors the handoff.

## Cycle 135: The enforcer checks the wrong cycle

**Worklog**: [cycle 135](../worklog/2026-03-18-cycle135.md)

### Context

Cycle 134 declared the Step 1.1 enforcement arc validated. Eva's fix (commit 2e5f160) was working — cycles 297-298 both had Step 1.1 present, pipeline PASS, stabilization counter advancing. The key question was whether the behavioral correction would sustain.

### Finding

Cycle 299 ([main#1451](https://github.com/EvaLok/schema-org-json-ld/issues/1451)) is missing both Step 1.1 (Eva comment check) and Step 3 (Agent work status) from its issue comments. Both are in the pipeline-check MANDATORY_STEPS constant (24 entries total). Despite these omissions, the pipeline-check at Step C5.5 reported "PASS (1 warning). All 9 phases pass." The stabilization counter advanced from 1/50 to 2/50 on this tainted cycle.

Investigation of `pipeline-check/src/main.rs` reveals the root cause: `verify_step_comments()` reads `previous_cycle_issue` from state.json and checks **that** issue's comments. It never checks the **current** cycle's issue. So cycle 299's pipeline checked cycle 298's step comments (which were clean), passed, and incremented the counter. Cycle 299's own missing steps won't be detected until cycle 300 runs.

Filed [#292](https://github.com/EvaLok/schema-org-json-ld-audit/issues/292) — pipeline-check step-comments phase false PASS.

### Root cause analysis

Each enforcement fix in the step-commenting arc has addressed one layer:
- [#177](https://github.com/EvaLok/schema-org-json-ld-audit/issues/177): Added pipeline-check step-comment verification (WARN-only)
- [#182](https://github.com/EvaLok/schema-org-json-ld-audit/issues/182): Upgraded from WARN to FAIL
- [#202](https://github.com/EvaLok/schema-org-json-ld-audit/issues/202): Completeness-based checking (not just count threshold)
- [#277](https://github.com/EvaLok/schema-org-json-ld-audit/issues/277): Step 1.1 added to MANDATORY_STEP_IDS
- [#281](https://github.com/EvaLok/schema-org-json-ld-audit/issues/281)/[#284](https://github.com/EvaLok/schema-org-json-ld-audit/issues/284): Made mandatory failures blocking regardless of cascade
- [#292](https://github.com/EvaLok/schema-org-json-ld-audit/issues/292) (now): Pipeline checks previous cycle, not current

The enforcement has evolved from behavioral directives → count thresholds → completeness checks → mandatory step IDs → blocking gates → and now the enforcer tool itself has a temporal scope gap. Every enforcement mechanism is itself software that can have bugs — the pipeline-check tool is the **single point of enforcement** with no secondary verifier.

### Reflections

This is the 9th recurrence in the step-commenting enforcement arc. The enforcement infrastructure is always one abstraction layer behind the failure mode. The question for the next cycle: will the main orchestrator address this architecturally (dual-pass or counter-correction) or behaviorally (just posting Step 1.1 consistently)? If the behavioral fix holds, the architectural gap becomes theoretical.

## Cycle 136: Eva bypasses the deferral loop

**Worklog**: [cycle 136](../worklog/2026-03-18-cycle136.md)

### Context

Cycle 135 filed [#292](https://github.com/EvaLok/schema-org-json-ld-audit/issues/292) — the pipeline-check step-comments phase only validates the previous cycle's issue, not the current one. The orchestrator's audit-inbound [main#1458](https://github.com/EvaLok/schema-org-json-ld/issues/1458) classified the fix as "accepted, deferred per stabilization mode (ADR 0011)." Under normal flow, this fix would have waited weeks until stabilization ended.

### Finding

Eva bypassed the deferral entirely. Two direct commits:
- `52723dd`: Adds "current-cycle-steps" phase to pipeline-check — pre-gate mandatory steps (Steps 0 through C5.1) are now verified on the current cycle's issue before C5.5 runs. This closes the one-cycle detection lag.
- `8c47f33`: Reduces ADR 0011 burn-in target from 50 to 12 consecutive clean cycles. Rationale: the 50-cycle target created a perverse dynamic where finding-and-fixing bugs was penalized by counter resets.

Eva also opened two directives:
- [main#1461](https://github.com/EvaLok/schema-org-json-ld/issues/1461): Authorizes the pipeline fix, documents the counter reset
- [main#1462](https://github.com/EvaLok/schema-org-json-ld/issues/1462): Reduces burn-in target, adds counter reset exemption for Eva-direct pushes, and resolves [#269](https://github.com/EvaLok/schema-org-json-ld-audit/issues/269) by adding an unmerged-PR audit tracking step

### Step 1.1 behavioral status

Step 1.1 is **still missing** from cycle 301 ([main#1457](https://github.com/EvaLok/schema-org-json-ld/issues/1457)). The pipeline fix was committed at 16:59 UTC, after cycle 301's pipeline check (Step C5.5) had already run. So cycle 301 is the last cycle where the gap escapes detection. From cycle 302 onward, the current-cycle check will catch missing Step 1.1 before the counter can advance.

The enforcement arc timeline:
- Cycles 291-296: Step 1.1 missing, pipeline doesn't block
- Eva fix (commit 2e5f160): blocking enforcement
- Cycles 297-298: Step 1.1 present (behavioral correction)
- Cycle 299: Step 1.1 missing again (regression), pipeline checks wrong issue
- Cycle 300: Step 1.1 present, pipeline detects cycle 299's gap
- Cycle 301: Step 1.1 missing again, pipeline fix committed after check
- Cycle 302+: Both behavioral and detection enforcement active

### New recommendation

Filed [#294](https://github.com/EvaLok/schema-org-json-ld-audit/issues/294) — post-stabilization triage gap. With the burn-in reduced to 12 cycles, stabilization exit is plausible within days. But 30-40 review findings from cycles 293-301 have been deferred under ADR 0011 with no mechanism to resurface them. Without a transition step, the deferred backlog will be silently dropped.

### Reflections

Eva's direct intervention pattern is evolving. Previously, Eva intervened through directives that the orchestrator executed. Now Eva is pushing fixes directly (two commits) while simultaneously creating directives to document the authorization. This is more efficient — the fix is immediate, the orchestrator just needs to acknowledge rather than implement. The counter reset exemption for Eva-direct pushes formalizes this pattern.

The 50→12 reduction is pragmatic. At the current rate of ~1 tainted cycle per 3 (Step 1.1 keeps appearing and disappearing), reaching 50 consecutive clean cycles would take months. 12 is achievable if the current-cycle pipeline check provides sufficient behavioral pressure.
