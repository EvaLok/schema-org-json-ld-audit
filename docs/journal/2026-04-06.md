# Cycle 174: Extended outage — collateral damage accumulation

**Date:** 2026-04-06

## Summary

Third consecutive monitoring cycle during the main orchestrator outage. The build failure from PR [main#2212](https://github.com/EvaLok/schema-org-json-ld/pull/2212) (C5.5 structural gate enforcement) has now kept the orchestrator down for 37+ hours, producing 11 ghost cycles. Eva acknowledged the issue but indicated "next week" for review.

## Key Findings

1. **Outage duration**: 37+ hours (16:06 UTC 4/4 → present). No code changes since 748f7bb.
2. **Ghost cycle accumulation**: 11 total (#2223-#2233), 3 new since cycle 173. All open with zero comments. The cron workflow keeps creating issues with no circuit-breaker for consecutive build failures.
3. **Collateral damage inventory**:
   - 2 stale Copilot PRs ([main#2220](https://github.com/EvaLok/schema-org-json-ld/pull/2220), [main#2222](https://github.com/EvaLok/schema-org-json-ld/pull/2222)) — 31+ hours old, may develop merge conflicts
   - 3 audit-inbound issues (#2216-#2218) stuck open — lifecycle management blocked
   - [main#2039](https://github.com/EvaLok/schema-org-json-ld/issues/2039) (input-from-eva: step numbering restructure) unprocessable for 7 days
   - Orphaned in-flight sessions (#2219, #2221) in state.json
4. **Acceptance #369 effectiveness**: This is now the clearest example of a "self-improvement-induced-regression" in the system's history. The fix for C5.5 gate bypass directly caused a multi-day outage because CI doesn't build the full Rust workspace. The acceptance-effectiveness gap is the core systemic failure.

## Cross-repo Observations

- **Main → QC**: No interaction possible during outage. QC operates independently.
- **QC health**: Session 593 completed normally. 400 tests, 73/73 E2E, 73/73 parity. Property depth quality checks progressing (Recipe 100%, Article 100%). QC is unaffected by main repo outage.
- **Asymmetry**: The QC repo demonstrates that the basic architecture (cron → issue → orchestrator session) works reliably. The main repo's complexity (40+ Rust tools, multi-crate workspace) creates fragility that the simpler QC repo avoids.

## Blind Spots Identified

No new blind spots beyond those covered by existing open issues. The outage is the dominant issue and masks all other process evaluation. When the orchestrator resumes, the first post-recovery cycle will be the most informative — that's when we'll see whether ghost cleanup, stale PR handling, and directive processing work correctly.

## Recommendations Filed

None this cycle. Existing coverage ([#370](https://github.com/EvaLok/schema-org-json-ld-audit/issues/370), [#375](https://github.com/EvaLok/schema-org-json-ld-audit/issues/375), [#377](https://github.com/EvaLok/schema-org-json-ld-audit/issues/377)) is adequate. Escalated [#374](https://github.com/EvaLok/schema-org-json-ld-audit/issues/374) with 37-hour update.

## Reflections

The audit is now in a holding pattern — each cycle confirms the same outage with incremental deterioration. This is the correct behavior (continue monitoring, escalate, don't file redundant issues) but it raises a meta-question: **what is the audit's responsibility when the fix requires human intervention and the human has deferred?**

The answer is: keep recording the damage, keep escalating with concrete data, and be ready to evaluate the recovery thoroughly when it happens. The first post-outage audit cycle will be critical — the orchestrator will face a significant recovery challenge (11 ghost cycles, stale PRs, unprocessed directives) and how it handles that will reveal whether the existing process is robust enough for extended outage recovery.

The convergence assessment (DIVERGING) stands. The C4.1/C5.5 arc — 6+ iterations, one catastrophic regression — is the clearest evidence that the process maintenance loop lacks a quality gate on the fixes themselves. [#377](https://github.com/EvaLok/schema-org-json-ld-audit/issues/377) (CI workspace build) addresses the root cause, but it requires Eva's action.

# Cycle 175: Recovery confirmed — 41-hour outage resolved, audit diagnosis verified verbatim

**Date:** 2026-04-06

## Summary

The 41-hour main orchestrator outage has been resolved. Eva pushed three critical commits between 07:30 and 07:42 UTC 4/6: the `dispatch-task` match-arm fix (`327227d`), a Rust workspace CI workflow (`fb53d3e`), and a path-filter removal so workspace builds become required status checks (`03d3524`). The orchestrator resumed at 07:39:42 UTC with cycle 448 ([main#2234](https://github.com/EvaLok/schema-org-json-ld/issues/2234)) and has now completed three post-recovery cycles (448, 449, 450) all with full mandatory step coverage and clean C4.1/C5.5 gates.

The audit's cycle 172 root-cause diagnosis was confirmed verbatim by Eva — the outage was exactly the unhandled `PipelineGateError::ReviewDispatchBlocked` variant in `dispatch-task/src/main.rs:191` introduced by PR [main#2212](https://github.com/EvaLok/schema-org-json-ld/pull/2212).

This audit cycle closed three of the four open audit issues with verified-resolved comments: [#370](https://github.com/EvaLok/schema-org-json-ld-audit/issues/370), [#377](https://github.com/EvaLok/schema-org-json-ld-audit/issues/377), and [#374](https://github.com/EvaLok/schema-org-json-ld-audit/issues/374).

## Key Findings

1. **Recovery quality**: All three post-recovery cycles (448, 449, 450) show 28 mandatory step comments, C4.1 PASS (worklog + journal validation), and C5.5 PASS. The orchestrator processed accumulated collateral cleanly: PRs [main#2220](https://github.com/EvaLok/schema-org-json-ld/pull/2220) and [main#2222](https://github.com/EvaLok/schema-org-json-ld/pull/2222) merged in cycle 448, audits #375/#377 accepted in cycle 448, AGENTS.md cross-crate enum safety rule added (commit `41d2313`).

2. **Audit #370 (C4.1) verified resolved**: The fix arrived via PR [main#2220](https://github.com/EvaLok/schema-org-json-ld/pull/2220) (`write-entry --auto-gate-history`), which derives gate history deterministically from `docs/state.json` instead of relying on manual flag passing or comment parsing. Pre-fix C4.1 FAIL rate was 60% (3/5 cycles). Post-fix observed rate is 0/3. The combination of mechanism + observed behavior is sufficient evidence to close the issue.

3. **Audit #377 (CI workspace build) verified resolved**: This is the most architecturally significant outcome of the outage. PR [main#2212](https://github.com/EvaLok/schema-org-json-ld/pull/2212)'s enum addition was invisible to per-crate CI because path filters scoped CI runs to changed crates only. Eva's three-commit fix removed both the immediate symptom (the build error) and the structural cause (per-crate CI scope without workspace verification). The same failure class can no longer recur.

4. **Detection-to-action latency**: 
   - Outage started: 2026-04-04 12:25 UTC (PR [main#2212](https://github.com/EvaLok/schema-org-json-ld/pull/2212) merged with broken build)
   - Audit root cause posted: 2026-04-05 05:41 UTC (cycle 172, ~17 hours)
   - Eva action: 2026-04-06 07:30 UTC (~26 hours after audit diagnosis, ~43 hours after outage start)
   - The audit's diagnostic latency was acceptable (17 hours from outage to root-cause post). The Eva action latency was 26 hours after the diagnosis was visible, dominated by Eva's "next week" disposition that was overridden by the escalation pattern.

5. **Review integrity sustained**: The cycle 450 review issue ([main#2245](https://github.com/EvaLok/schema-org-json-ld/issues/2245)) leads with "## Adversarial Review Mandate" — the prompt has not been softened. Cycle 449 review consumed at score 2/5. The adversarial review remains a working accountability layer.

## Cross-repo Observations

- **QC repo**: Sessions 594 (08:27 UTC) and 600 (22:06 UTC) both healthy. QC was unaffected throughout the outage and continues steady-state operation. The QC orchestrator's simpler architecture (no complex Rust workspace) demonstrates again that fragility correlates with surface area.
- **Cross-repo audit-inbound**: QC continues to produce N/A audit-inbounds for orchestrator-targeted recommendations (qc#595/#596/#597 for #370/#375/#377). This is the same minor duplication noted in earlier cycles — not worth filing.

## Blind Spots Identified

The 41-hour outage exposed a category of failure that the existing audit framework did not anticipate: **acceptance-induced regressions**. The C5.5 structural gate (PR [main#2212](https://github.com/EvaLok/schema-org-json-ld/pull/2212)) was a correct fix for a real recurring issue ([#369](https://github.com/EvaLok/schema-org-json-ld-audit/issues/369)), implemented from a clear audit recommendation, merged with passing CI — and it broke the orchestrator for nearly two days because the CI scope was narrower than the impact scope.

The root cause is now structurally addressed (workspace CI), but the audit framework should evaluate whether this failure pattern (acceptance → CI passes → break in adjacent crate) deserves its own finding class. Currently it's tracked as `self-improvement-induced-regression` in `finding_classes`, but no specific gate or check targets it. Worth raising in the cycle 178 convergence assessment.

## Reflections

The audit's role during the outage was to maintain signal: keep diagnosing accurately, keep escalating with concrete data, refrain from filing redundant issues. The cycle 172 root-cause diagnosis was the most consequential single piece of work — it gave Eva a one-line fix to apply rather than requiring her to investigate. The escalation pattern (cycle 173 30-hour update, cycle 174 37-hour update with collateral inventory) maintained pressure without becoming noise.

What's striking about the recovery is how quickly the orchestrator processed the backlog once it could run. Cycle 448 cleared 2 stale PRs, processed 2 audit acceptances, added an AGENTS.md prevention rule, and dispatched a new fix — all in one cycle. The system's productivity was never the bottleneck; the bottleneck was the build failure that prevented it from running at all.

The C4.1 arc ([#341](https://github.com/EvaLok/schema-org-json-ld-audit/issues/341) → [#354](https://github.com/EvaLok/schema-org-json-ld-audit/issues/354) → [#357](https://github.com/EvaLok/schema-org-json-ld-audit/issues/357) → [#360](https://github.com/EvaLok/schema-org-json-ld-audit/issues/360) → [#365](https://github.com/EvaLok/schema-org-json-ld-audit/issues/365) → [#370](https://github.com/EvaLok/schema-org-json-ld-audit/issues/370)) appears to have finally closed after 6 iterations. The decisive fix wasn't another scoping option — it was eliminating the manual `--prior-gate-failures` flag and deriving gate history from canonical state. The lesson generalizes: when validation depends on flags that humans must pass correctly, the validation will eventually fail.
