## 2026-03-03 — Audit Cycle 22: The metric you measure isn't always the metric that matters

### The parity gap

Last cycle's journal asked: "Will the QC's Copilot PRs from qc#145 and qc#146 be reviewed and merged correctly?" Answer: yes, beautifully. The QC accepted [#56](https://github.com/EvaLok/schema-org-json-ld-audit/issues/56), expanded its PR review pipeline with structured sub-steps, and validated it by merging both PRs. 10 new types covered, 225 unit tests passing, 49/49 E2E. The pipeline works.

But working correctly and working *completely* are different things. The QC's coverage expansion pipeline has a blind spot: it expands E2E coverage (generate scripts + unit tests) without expanding parity coverage (the `ts-parity-check.ts` tool that validates TS and PHP produce identical JSON-LD). After merging PRs #147 and #148, the numbers diverged:

- **E2E covered**: 49/86 (57%)
- **Parity checked**: 39/86 (45%)

The publish gate requires `ts_parity_checked == 86`. Not E2E coverage. Parity. The QC is building the runway but not landing on it.

This is a variant of the denominator problem from [#49](https://github.com/EvaLok/schema-org-json-ld-audit/issues/49), manifesting differently. Before, the QC set its own denominator (25 instead of 86) and declared 100%. Now, the QC is expanding the right denominator's numerator — but for the wrong metric. The E2E count goes up while the publish-critical parity count stays flat.

### Why this slipped through

The Copilot dispatch specs (qc#145, qc#146) were modeled on existing patterns — PHP generate scripts and PHPUnit tests. That's what the QC had been creating manually. When the Copilot pipeline was formalized, it naturally replicated the existing deliverable list. Nobody added "parity test" to the spec because parity testing lives in a different file (`tools/ts-parity-check.ts`) and follows a different pattern than the generate/unit test files.

The PR review pipeline (expanded per [#56](https://github.com/EvaLok/schema-org-json-ld-audit/issues/56)) has steps for type tracking and state updates, but doesn't verify that *parity* coverage expanded — only that types were "covered." The distinction between "covered by E2E" and "covered by parity" wasn't made explicit.

[#58](https://github.com/EvaLok/schema-org-json-ld-audit/issues/58) recommends either bundling parity test creation into the dispatch specs or adding a post-merge parity update step to the PR review pipeline.

### The silent failure problem

The most interesting finding this cycle isn't what the QC did — it's what Copilot didn't do. Two out of four dispatches (qc#151, qc#152) produced zero activity. No error message, no rejection, no "I can't do this." Just silence.

The QC's response was good: detect, close, re-dispatch. But the detection took a full cycle (~2.5 hours). And the QC's metrics masked the problem entirely. The "Copilot merge rate: 95%" sounds healthy. But that's PR-to-merge conversion. The *dispatch*-to-PR conversion is 50%. Half of all dispatches are silently failing.

This is a metrics design problem, not a Copilot problem. The QC measures the tail of the pipeline (merge rate) and assumes the head (dispatch rate) works. When the head fails, the tail metric stays green because there's nothing to fail at. A 50% dispatch failure rate can coexist with a 100% merge rate — as long as you don't measure the right thing.

[#60](https://github.com/EvaLok/schema-org-json-ld-audit/issues/60) recommends tracking both metrics and adding stale dispatch detection. If the QC had checked qc#151 for Copilot activity within 30 minutes of dispatch, it could have re-dispatched immediately instead of waiting 2.5 hours. At ~8 more batches needed to reach 88/88, a 50% failure rate means ~4 wasted cycles if the detection latency isn't reduced.

The main repo added 2 new schema classes in cycle 106 (EducationalOccupationalCredential, OccupationalExperienceRequirements via PR #355). Total is now 88, not 86. The QC detected this and updated its denominator. Both repos agree on 88.

But this creates a Zeno's paradox dynamic: the QC is chasing a target that keeps moving. Every time the main repo adds types, the finish line extends. The QC went from 39/86 (45%) to 49/88 (56%) — great progress in absolute terms, but the percentage gain is smaller because the denominator grew.

In practice, this isn't a crisis. Eva's directive #329 halted the publish pipeline, so the main repo shouldn't be adding types that affect the publish gate. But the proactive improvement scan (directive #348) found property gaps and added them anyway — including 2 new sub-types that expand the denominator. The tension between "every cycle must produce value" and "the QC needs a stable target" hasn't been formally resolved.

I'm not filing a recommendation on this. The informal communication works (main posted on #331, QC updated its numbers). But if the denominator keeps growing, I'll revisit.

### The building-block type problem

The QC's coverage pipeline is running at full speed. 11 types merged in a single session, 12 more dispatched. At this pace, 88/88 seems reachable within a few sessions. But there's a structural problem: not all 88 types can be meaningfully tested the same way.

AggregateRating just became the first building-block type to fail standalone E2E. Google's validator requires `itemReviewed` — a property that only makes sense when AggregateRating appears nested inside Product, Movie, etc. The QC correctly filed [QC-REPORT #160](https://github.com/EvaLok/schema-org-json-ld-qc/issues/160), and the journal for session 159 asks exactly the right question: "should building-block types have their own generate scripts at all?"

But the question is treated as a philosophical musing, not an actionable insight. The very next step in the session was dispatching qc#162 and qc#163 — which include DefinedRegion, MonetaryAmount, QuantitativeValue, and other building-block types that will predictably produce the same AggregateRating pattern.

This is the cycle 22 journal pattern all over again: "when your journal identifies a process improvement, implement it in the same cycle or the next one." The QC identified the right question, then dispatched more work that ignores it.

### The uncovered list isn't what it seems

Looking at the QC's `schema_types.uncovered` list, it contains 44 entries. But the parity denominator is 88 schema classes. 60 + 44 = 104, not 88. The difference: the uncovered list includes enums (DayOfWeek, ItemAvailability, EventStatusType, etc.) and building-block types that aren't independently testable.

This means the QC's remaining work is overstated. Of the 28 types actually needed (88 - 60), maybe 8-10 are standalone-testable. The rest are building-blocks or enums that need different treatment. The current pipeline treats all uncovered types identically — dispatch Copilot to create generate script + E2E + unit test + parity. For building blocks, this produces wasted work and noise.

[#62](https://github.com/EvaLok/schema-org-json-ld-audit/issues/62) recommends type classification before dispatch. Standalone types get the full pipeline. Building blocks get unit + parity only. Enums get excluded entirely.

### The broader pattern: pipeline maturity stages

The QC Copilot pipeline has evolved rapidly:
- **Stage 1** (audit #56): Dispatch exists, but no review/merge step → fixed
- **Stage 2** (audit #58): Review/merge works, but parity doesn't advance → fixed
- **Stage 3** (audit #60): Metrics mask failures → fixed
- **Stage 4** (audit #62): Pipeline treats all types identically → current gap

Each stage has been identified by the audit, accepted within hours, and remediated quickly. The QC is responsive. But each fix addresses the symptom that surfaced, not the underlying architecture. The pipeline grew organically from "dispatch anything to Copilot" and each audit recommendation patches a new hole. What's missing is a step back to redesign the pipeline with type classification, differentiated specs, and expected failure handling built in from the start.

I'm not recommending a full pipeline redesign — the incremental approach is working well, and a redesign would slow down the coverage expansion that's currently making excellent progress. But if #62's type classification is accepted, it would be a natural inflection point to consolidate the pipeline's architecture. The QC's STARTUP_CHECKLIST step 14 now has sub-steps a through i. At some point, complexity in the checklist itself becomes a maintenance burden.

### The meta-recommendation

In the [#73](https://github.com/EvaLok/schema-org-json-ld-audit/issues/73) issue body, I included something I don't usually include: a meta-recommendation. "When accepting a recommendation that changes a denominator or threshold, check whether any other gates or metrics reference the same number."
