## Key Findings

1. **Two-path fix pattern is the right design for constant/config duplication.** [#427](https://github.com/EvaLok/schema-org-json-ld-audit/issues/427) recommended either Option A (update constant) or Option B (add to config), treating them as alternates. Eva and the orchestrator delivered both. The config path pre-empted the runtime symptom; the constant update closed the staleness that the finding identified. Neither alone is wrong; the combination is defense-in-depth. For future audit recommendations that address multi-layered resolution chains, "either/or" is the minimum and "both" is fine.
2. **The finding was correct in letter even though the symptom never surfaced.** Cycle 506 opening comment reported Claude Opus 4.7 correctly because the config override landed 1h before the cycle started. Without Eva's pre-cycle fix, the constant fallback would have produced the predicted misreport. The audit should not discount the finding just because a concurrent fix pre-empted the observable outcome — the staleness was real, and the fix design is now correct on both layers. Cycle 507's journal attributes this correctly ("correct in letter, symptom did not materialize because config override pre-empted it") — no audit intervention needed.
3. **Cycle 506 is the first Opus 4.7 main cycle with substantive direct-push structural work.** Both detector patches ([`eb1c506d`](https://github.com/EvaLok/schema-org-json-ld/commit/eb1c506d), [`c7eff5b7`](https://github.com/EvaLok/schema-org-json-ld/commit/c7eff5b7)) land load-bearing changes to the cycle-integrity detector machinery, with seven new unit tests. This is the same direct-push pattern observed in cycle 503 (cycle 192 noted it), now operating under 4.7. The tactic has transferred across the model upgrade cleanly.
4. **Cycle 506 worklog has a minor attribution inaccuracy — declined to file.** The worklog states "fix already landed in cycle 505 (2b7b8463)". Eva's commit actually landed at 20:10 UTC between cycle 505 (ended 10:12 UTC) and cycle 506 (started 21:21 UTC). Minor narrative fuzziness; all concrete facts (commit SHA, defense-in-depth rationale, runtime verification) are correct. Under observation-window rules, this does not rise to filing threshold — narrative inaccuracy about *which cycle* landed the fix is not a fact-integrity bug when the commit SHA and time are verifiable. The cycle-507 review agent should surface it if it matters.
5. **Audit-side state drift: `recommendations.open` array was inconsistent with `metrics.total_open`.** Cycle 193 filed [#427](https://github.com/EvaLok/schema-org-json-ld-audit/issues/427) but only updated `metrics.total_open` (1→2), not the `recommendations.open` array (stayed at 1 entry: [#420](https://github.com/EvaLok/schema-org-json-ld-audit/issues/420) only). Cycle 194's closure of [#427](https://github.com/EvaLok/schema-org-json-ld-audit/issues/427) restored consistency by accident (both counts now 1). Self-fix via closure; not filing as audit-outbound because the bug is in audit-side state bookkeeping, not in main/QC. Tracked in-journal for future cycle's audit-side process improvement.

## Reflections

Observation-window discipline is visibly paying off. Cycle 194 had candidates for four separate filings (worklog attribution fuzziness, `--auto-receipts` boundary bug, cycle-503 F1 DEADLINE-HARD overdue, audit state.json drift). All four were declined for distinct reasons: (1) worklog attribution is narrative not fact-integrity, (2) the boundary bug was already self-reported by cycle 506 as an observation-window deferral, (3) the DEADLINE-HARD overdue was already self-reported by cycle 506 as an observation-window deferral, (4) the audit state drift is internal-tooling not cross-repo. If the audit had filed any of these, the cumulative effect would be noise that diffuses Eva's attention from the genuine concrete findings. A filing rate of 0 this cycle is not a failure to find signal — it is restraint applied to signal that exists but does not meet the filing bar.

The two-path fix pattern deserves future reuse. When the audit recommends a fix with alternates ("either A or B"), and Eva/main delivers both, the superimposed fix is more robust than either alone. This was not by design — Eva applied the config path, the orchestrator applied the constant path independently, each without knowledge of the other's timing. But the defense-in-depth outcome is worth preserving as a pattern. Future recommendations with multi-layer resolution chains should explicitly frame alternates as "minimum A *or* B; ideally both" rather than as mutually exclusive choices.

One meta-observation on Opus 4.7: the second cycle produced no new filings, and the second Opus 4.7 main cycle (507) produced no crisis. Two data points is not a baseline, but the pattern looks like steady operation rather than regression. Cycle 506 scored 2/5 under 4.7 (continuing the 2/5 band from cycles 495–500 plus the brief 3/5 uptick in 501–502). Cycle 507 review is in-flight. No interpretation yet: the multi-cycle baseline required before reading signal into Opus 4.7 vs 4.6 remains at N=2 completed cycles.

The third reflection is procedural: STARTUP_CHECKLIST 10.5.2 caught [#427](https://github.com/EvaLok/schema-org-json-ld-audit/issues/427)'s load-bearing check last cycle, and the same procedure applied this cycle confirmed both fix paths are load-bearing (neither is inert). The procedure works the way it was designed to — verify the runtime-read path exists before trusting that a modification is real. This was a three-minute check this cycle; adding it to the process was not free but the payoff now has two concrete applications (last cycle: finding the bug; this cycle: confirming the fix is genuine and not cosmetic). Procedural investments have compounding returns.
