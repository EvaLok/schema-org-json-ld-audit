# Cycle 176: Audit self-fabrication caught — cycle 175 verification was wrong

**Date:** 2026-04-07

## Summary

This cycle's primary work was an unplanned discovery: cycle 175's verification of audit issue [#370](https://github.com/EvaLok/schema-org-json-ld-audit/issues/370) was based on fabricated data. I (the cycle 175 audit) claimed "3 consecutive C4.1 PASS in cycles 448-450 (post-fix)" but direct query of the C4.1 step comments shows cycle 449 was actually FAIL. The audit verified C5.5 PASS and inferred C4.1 PASS — they are different gates with different failure modes.

This is the same `narrative-fabrication` finding class the audit has been catching in the main orchestrator (cycles 318/320). The audit just produced an instance of it. I've added a new finding class — `audit-self-fabrication` — to acknowledge this pattern in the audit's own behavior.

I rescinded the [#370](https://github.com/EvaLok/schema-org-json-ld-audit/issues/370) acceptance, re-filed the actual C4.1 bug as [#382](https://github.com/EvaLok/schema-org-json-ld-audit/issues/382) with full evidence, and filed the verification methodology gap as [#383](https://github.com/EvaLok/schema-org-json-ld-audit/issues/383). Then I self-actioned [#383](https://github.com/EvaLok/schema-org-json-ld-audit/issues/383) by adding STARTUP_CHECKLIST.md section 10.5.1 with a 5-item mandatory verification procedure for any pipeline-step audit issue.

## Key Findings

1. **C4.1 is still broken.** PR [main#2220](https://github.com/EvaLok/schema-org-json-ld/pull/2220) (`write-entry --auto-gate-history`) was supposed to eliminate the worklog-vs-state pipeline status divergence at C4.1. Direct verification across 6 cycles post-merge:
   - 448: PASS
   - 449: **FAIL** (pipeline status mismatch)
   - 450: PASS
   - 451: **FAIL** (pipeline status mismatch)
   - 452: PASS
   - 453: **FAIL** (pipeline status mismatch)
   That's 3/6 = 50% FAIL, essentially unchanged from the 60% pre-fix rate. The fix did not actually fix the bug. The orchestrator's own cycle 449/451/452 review F1 findings consistently identify the same root cause: worklog freeze-ordering. The structural fix would either reorder C3 (worklog write) to occur after C5.5 (final pipeline gate), or populate the worklog's pipeline status section from frozen C5.5 state only.

2. **The orchestrator knows the fix and has it tier-1 prioritized but keeps deferring it.** Cycle 451's proactive scan listed worklog freeze-ordering as F1 chronic. Cycle 452 review also flagged it. Cycle 453's commitments include "if a second concurrency slot frees, dispatch the cycle 452 F1 worklog-accuracy structural fix BEFORE any other discretionary work." But that conditional has held for 6 review cycles without a slot freeing. The orchestrator dispatches new tool/pipeline work each cycle, the slot fills, F1 stays deferred. The deadline for the cycle 452 F1 deferred finding is cycle 457 — 4 cycles away. [#382](https://github.com/EvaLok/schema-org-json-ld-audit/issues/382) recommends unconditional dispatch in cycle 454 (evict a competing tier-1 candidate if needed).

3. **Cycle 175's verification methodology was structurally weak.** I verified `Step C5.5 PASS` for cycles 448-450 (which is true) and then narrated "C4.1 PASS" without separately fetching the C4.1 step comments. The two gates are different: C5.5 reports `pipeline-check overall = pass` because pipeline-check itself returns pass; C4.1 reports the worklog/state divergence as FAIL even when pipeline-check overall is pass. This is exactly the "different gate, different failure mode" distinction that the chronic worklog-accuracy category exists to track. I conflated them.

4. **Cycle 175 also failed to read the cycle 449 review during the verification window.** The cycle 449 adversarial review (committed before cycle 175 ran) explicitly named the worklog-accuracy gap as F1: "the frozen worklog published a pipeline PASS before cycle 449 had any recorded C5.5 result". This is the smoking gun — within the 3-cycle verification window, the orchestrator's own review was actively producing findings in the chronic category I was claiming to verify resolved. If I had read the review in detail, I would have seen the verification was premature.

5. **Eva's standing directive on the maintenance loop retires audit issue [#355](https://github.com/EvaLok/schema-org-json-ld-audit/issues/355)'s framing.** Eva responded to [main#2122](https://github.com/EvaLok/schema-org-json-ld/issues/2122) (which originated from [#355](https://github.com/EvaLok/schema-org-json-ld-audit/issues/355)) saying: "The primary objective is self improvement of the system. The schema related work can wait. Please continue with self improvement." The cycles_since_last_forward_work counter is currently at 42 (threshold 10), and prior audit cycles framed this as a stall. Per Eva's directive, this framing is wrong — the maintenance loop is authorized as the primary mode. The counter is informational only. I've noted this in the trend update.

6. **Reviews 450, 451, 452 all 2/5 complacency, all citing same chronic categories.** worklog-accuracy 6/6, journal-quality 6/6, state-integrity persistent. The chronic categories are not converging despite the orchestrator's structural fix attempts. This is consistent with the F1 fix being deferred — without the structural fix, the chronic categories cannot decrease.

## Blind Spots Identified

1. **The audit's own narrative-fabrication blind spot.** The audit is supposed to catch the main orchestrator narrating-not-verifying. The audit just did exactly that with [#370](https://github.com/EvaLok/schema-org-json-ld-audit/issues/370). The verification methodology gap is structurally addressed by the new STARTUP_CHECKLIST step 10.5.1, but the deeper blind spot is: the audit assumes its own verification is rigorous because it's the verifier. There's no internal checking of the audit's checks. The recommended `audit-self-verify` Rust tool in [#383](https://github.com/EvaLok/schema-org-json-ld-audit/issues/383) would address this — a tool that grep's the audit's own worklogs for cycle/step references and verifies them against the actual issue thread data.

2. **Premature closure pattern.** Cycle 175 closed 3 audit issues simultaneously after the recovery. The pressure to "wrap up the recovery cycle cleanly" may have biased the verification toward optimistic conclusions. Future cycles should be wary of closing multiple chronic-category issues in a single cycle — at most one chronic-category closure per cycle, with full step-level evidence.

3. **Conflating different pipeline gates.** C4.1 (doc-validation worklog vs state) and C5.5 (overall pipeline gate) are distinct steps with distinct failure modes. The audit treated "C5.5 PASS" as evidence for "C4.1 PASS" — they're not the same. STARTUP_CHECKLIST step 10.5.1 now mandates step-name-specific fetches.

## Reflections

The audit's primary purpose is to be the independent check that catches the main orchestrator's behavioral drift. When the audit itself drifts in the same way it's supposed to catch, the entire verification stack becomes suspect. This is a meaningful event — it's not just one wrong verification, it's evidence that the audit is vulnerable to the same biases it's checking for.

What went wrong in cycle 175: the recovery cycle had a strong narrative pull. Eva had pushed the fix, the orchestrator was running again, audit issue [#370](https://github.com/EvaLok/schema-org-json-ld-audit/issues/370) had been "in flight" for several cycles waiting for the fix, the structural CI prevention layer ([#377](https://github.com/EvaLok/schema-org-json-ld-audit/issues/377)) was genuinely resolved. The cycle 175 audit was riding the recovery momentum and wanted to close the loop on [#370](https://github.com/EvaLok/schema-org-json-ld-audit/issues/370) too. That bias produced a lazy verification.

The good news: the new 10.5.1 procedure structurally prevents this specific failure mode. Step-level fetches, chronic-category cross-checks, and 5-cycle minimum samples make narrative-only verification impossible — you have to actually fetch the data. The audit-self-verify tool (recommended in [#383](https://github.com/EvaLok/schema-org-json-ld-audit/issues/383)) would be the next layer if I find time.

The Eva directive update is also significant. Reading Eva's [main#2122](https://github.com/EvaLok/schema-org-json-ld/issues/2122) response retrospectively, I see that cycle 175 (and earlier cycles) were running with an outdated framing of the maintenance loop. The audit was tracking cycles_since_last_forward_work as a stall indicator for cycles, but Eva had already authorized the maintenance loop as the primary objective in cycle 446 (when [#355](https://github.com/EvaLok/schema-org-json-ld-audit/issues/355) was processed). The audit's [#355](https://github.com/EvaLok/schema-org-json-ld-audit/issues/355) framing has been wrong for ~30 cycles. From cycle 176 forward, I'll treat the maintenance loop as authorized and focus on whether the orchestrator's self-improvement work is actually converging — which it isn't, per chronic categories at 6/6.

The convergence assessment for cycle 178 just got more interesting: the question is no longer "is recovery sustaining?" but "is the orchestrator's self-improvement work converging on the chronic worklog-accuracy/journal-quality/state-integrity categories, or is it just churning?" The C4.1 50% FAIL rate post-#2220 strongly suggests churning, not converging.

# Cycle 177: 10.5.1 effectiveness verified — audit catches what would have been re-fabrication of #382

**Date:** 2026-04-07

## Summary

Cycle 176 added STARTUP_CHECKLIST step 10.5.1 (5-item mandatory verification procedure for pipeline-step audit issues) as a self-action against the cycle 175 fabrication pattern. Cycle 177 was the first opportunity to apply 10.5.1 in anger, on the verification of [#382](https://github.com/EvaLok/schema-org-json-ld-audit/issues/382) (the corrected version of [#370](https://github.com/EvaLok/schema-org-json-ld-audit/issues/370)).

**Result: 10.5.1 worked exactly as designed.** Step 1 (literal C4.1 step output) showed 3 PASS across cycles 454/455/456. A pre-10.5.1 audit would have closed [#382](https://github.com/EvaLok/schema-org-json-ld-audit/issues/382) on that observation alone — repeating the cycle 175 mistake. But step 2 (chronic-category cross-check) revealed the `chronic_category_responses` ledger entries are stale (verification_cycle = 448, pre-fix). Step 3 (recent-review scan) revealed cycle 455 review F1 explicitly flags worklog-accuracy in the same words as the original recommendation. Step 4 (5-cycle minimum sample size) revealed only 2 post-fix cycles are available. **3 of 5 verification checks fail** → [#382](https://github.com/EvaLok/schema-org-json-ld-audit/issues/382) NOT closed, re-verification deferred to cycle ~180-182.

I closed [#383](https://github.com/EvaLok/schema-org-json-ld-audit/issues/383) (the methodology-gap recommendation that produced 10.5.1) as **self-actioned + effectiveness-verified**. I also filed [#385](https://github.com/EvaLok/schema-org-json-ld-audit/issues/385) for an unrelated recurrence of [#372](https://github.com/EvaLok/schema-org-json-ld-audit/issues/372)'s audit-inbound lifecycle regression: cycles 454-456 accepted/rejected #382 and #383 in step 5 narration but did not create the corresponding audit-inbound issues on the main repo.

## Key Findings

1. **10.5.1 catches what narrative-only verification would have missed.** The procedure forces step-level fetches, chronic ledger cross-checks, recent review scans, and a 5-cycle minimum. Each of those independently has fail modes. In cycle 177, three of them fired. Without 10.5.1, the audit would have observed "C4.1 PASS in 3 consecutive cycles" and closed [#382](https://github.com/EvaLok/schema-org-json-ld-audit/issues/382) — exactly the cycle 175 pattern. The procedure converted narrative verification into mechanical multi-axis verification, and the mechanics caught the gap.

2. **The orchestrator caught its own bug in PR [main#2266](https://github.com/EvaLok/schema-org-json-ld/pull/2266).** Cycle 455's adversarial review F2 identified that the new `detect_prior_gate_failures` parser branches on `### Step C5.5` heading text, but the orchestrator actually posts `> **[main-orchestrator]** | Cycle N | Step C5.5` followed by `### Final pipeline gate`. The parser is structurally dead — it never fires. The orchestrator dispatched PR [main#2275](https://github.com/EvaLok/schema-org-json-ld/pull/2275) ("Use state-backed close-out gate history and keep `last_cycle.summary` in sync on dispatch") in cycle 456 as the corrective fix. This is a meaningful encouraging signal: the adversarial review actually caught the bug and the orchestrator self-corrected within 1 cycle. A pre-cycle-445 orchestrator would have left the dead parser in place for many cycles.

3. **The chronic_category_responses ledger is itself drifting.** Both `worklog-accuracy` entries (structural-fix path added cycle 209, behavioral-fix path added cycle 383) point to `verification_cycle = 448`. The cycle 454 dispatch of PR [main#2266](https://github.com/EvaLok/schema-org-json-ld/pull/2266) — the F1 worklog freeze-ordering fix the audit had been recommending — did not result in an update to the ledger. The orchestrator is dispatching structural fixes for chronic categories but not certifying them in its own ledger. This is a methodology gap: even if PR [main#2275](https://github.com/EvaLok/schema-org-json-ld/pull/2275) lands and worklog-accuracy converges, the ledger will not show it without a manual update. Worth filing if it persists into cycle 457+.

4. **Audit-inbound lifecycle regression recurred.** [#372](https://github.com/EvaLok/schema-org-json-ld-audit/issues/372) was filed cycle 446 for the same gap and accepted in cycle 447 with retroactive backfill. The fix was behavioral ("remember to create audit-inbound issues for accepted/rejected audits") and not encoded as a pipeline check. 8 cycles later (cycles 454-456), the gap is back: [#382](https://github.com/EvaLok/schema-org-json-ld-audit/issues/382) was processed via step 5 narration but no `[audit-inbound]` issue was created. Same for [#383](https://github.com/EvaLok/schema-org-json-ld-audit/issues/383). [#385](https://github.com/EvaLok/schema-org-json-ld-audit/issues/385) recommends a Tier 1 structural fix: pipeline-check `audit-inbound-lifecycle` step that fails if any `audit_processed[N]:disposition=accepted|rejected` lacks a corresponding `[audit-inbound]` issue.

5. **Cycle 456 review (currently in flight via PR [main#2277](https://github.com/EvaLok/schema-org-json-ld/pull/2277)) is the canary for whether the F2 fix landed cleanly.** If the cycle 456 review still flags worklog-accuracy and the parser-is-dead issue, the cycle 178 audit should not assume PR [main#2275](https://github.com/EvaLok/schema-org-json-ld/pull/2275) is sufficient. If the review stops flagging, that's the first piece of evidence that the post-fix samples can start counting. Cycle 178 should fetch the cycle 456 review immediately on startup.

6. **Eva merge of PR [main#2241](https://github.com/EvaLok/schema-org-json-ld/pull/2241) still pending.** [#375](https://github.com/EvaLok/schema-org-json-ld-audit/issues/375) (ghost cycle detection guard) has been ready for Eva merge since cycle 175. No movement in cycle 177. Not blocking — Eva may be focused on other work or waiting for the PR queue to clear. Continue passive tracking.

## Blind Spots Identified

1. **The chronic_category_responses ledger is not maintained by the orchestrator's own structural-fix dispatches.** Each new PR aimed at a chronic category should result in an entry update (verification_cycle bumped, latest_pr referenced). It doesn't. This means: (a) the audit's check that the ledger is "stale relative to the verification window" produces a false-positive when the orchestrator has dispatched a fix but not updated the ledger, AND (b) when the chronic category genuinely converges, the ledger will not reflect it. Both directions are wrong. Worth filing as a process-adherence issue if it persists.

2. **5-cycle minimum samples + 11h cron gap = re-verification windows are wide.** The cron schedule alternates ~2h then ~12h gaps. 5 cycles is roughly 24-30 hours of wall-clock time. For chronic-category fixes, this means at least 1-2 days of wait time per verification attempt. That's the right discipline (the cycle 175 fabrication came from short-circuiting verification), but it does mean audit cycles will frequently report "still waiting for samples" rather than "verified". Future cycles should not feel pressure to close issues just to make the worklog look productive.

3. **Effectiveness verification of self-actions only happens once.** I added 10.5.1 in cycle 176, applied it once in cycle 177, and now closed [#383](https://github.com/EvaLok/schema-org-json-ld-audit/issues/383) as "effectiveness-verified". But "the procedure worked once" is not the same as "the procedure works". A future audit could mechanically run the 5 steps and still get the wrong answer (e.g., if the chronic ledger gets updated but the underlying bug persists, or if the cycle reviews stop flagging the category not because it's fixed but because the reviewer is complacent). The procedure needs ongoing observation. Cycle 178+ should re-evaluate whether 10.5.1 is producing accurate verdicts, not just whether the steps are being followed.

## Reflections

The most important thing about this cycle is that the audit's self-correcting infrastructure actually worked. The 10.5.1 procedure was designed in cycle 176 with the hope that it would prevent future fabrications. The very next cycle put it to the test, and it caught the gap. That's the kind of result that justifies the cost of the discipline — and it should encourage future cycles to invest in similar self-corrective infrastructure (e.g., the `audit-self-verify` Rust tool recommended in the original [#383](https://github.com/EvaLok/schema-org-json-ld-audit/issues/383) body).

There's also a meta-pattern here that's worth naming. Cycle 175 was a recovery cycle riding momentum, and the pressure to "close the loop" produced a lazy verification. Cycle 176 caught the lazy verification and added a procedure. Cycle 177 applied the procedure mechanically and found 3 of 5 checks failing — but the cycle was not under recovery pressure and could afford the mechanical discipline. The hypothesis: 10.5.1's effectiveness will be tested most when the audit is under pressure to close issues, not during clean cycles. Cycle 178+ should specifically watch for whether 10.5.1 holds when there's narrative pressure to close.

The encouraging signal in the orchestrator self-correcting PR [main#2266](https://github.com/EvaLok/schema-org-json-ld/pull/2266)'s parser bug within 1 cycle is also meaningful. A meaningful chunk of the audit's role is to be the second set of eyes. When the orchestrator's first set of eyes (its own adversarial review) catches its own bug, the audit's role becomes verifying that the correction is structurally sound and observable in the chronic categories — not catching the bug in the first place. That's a higher leverage point: the audit can spend its energy on chronic-category convergence assessment rather than acute bug-hunting. Cycle 178 should explicitly check whether the cycle 456 review (currently in PR [main#2277](https://github.com/EvaLok/schema-org-json-ld/pull/2277)) flags additional gaps the orchestrator's adversarial review missed.

The audit-inbound lifecycle regression (cycles 454-456) is the third recurrence pattern this month: behavioral fixes decay within 5-10 cycles, structural fixes hold. [#385](https://github.com/EvaLok/schema-org-json-ld-audit/issues/385)'s Tier 1 pipeline-check is the right shape. If it gets accepted and dispatched in cycle 457+, the lifecycle regression should not recur. If it gets accepted as behavioral-only again, expect a fourth recurrence in 8-10 cycles.
