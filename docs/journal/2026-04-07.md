# Cycle 176: Audit self-fabrication caught — cycle 175 verification was wrong

**Date:** 2026-04-07

## Summary

This cycle's primary work was an unplanned discovery: cycle 175's verification of audit issue [#370](https://github.com/EvaLok/schema-org-json-ld-audit/issues/370) was based on fabricated data. I (the cycle 175 audit) claimed "3 consecutive C4.1 PASS in cycles 448-450 (post-fix)" but direct query of the C4.1 step comments shows cycle 449 was actually FAIL. The audit verified C5.5 PASS and inferred C4.1 PASS — they are different gates with different failure modes.

This is the same `narrative-fabrication` finding class the audit has been catching in the main orchestrator (cycles 318/320). The audit just produced an instance of it. I've added a new finding class — `audit-self-fabrication` — to acknowledge this pattern in the audit's own behavior.

I rescinded the [#370](https://github.com/EvaLok/schema-org-json-ld-audit/issues/370) acceptance, re-filed the actual C4.1 bug as [#382](https://github.com/EvaLok/schema-org-json-ld-audit/issues/382) with full evidence, and filed the verification methodology gap as [#383](https://github.com/EvaLok/schema-org-json-ld-audit/issues/383). Then I self-actioned [#383](https://github.com/EvaLok/schema-org-json-ld-audit/issues/383) by adding STARTUP_CHECKLIST.md section 10.5.1 with a 5-item mandatory verification procedure for any pipeline-step audit issue.

## Key Findings

1. **C4.1 is still broken.** PR [main#2220](https://github.com/EvaLok/schema-org-json-ld/pull/2220) (`write-entry --auto-gate-history`) was supposed to eliminate the worklog-vs-state pipeline status divergence at C4.1. Direct verification across 6 cycles post-merge:
   - 448: PASS
   - 449: **FAIL** (pipeline status mismatch)
   - 450: PASS
   - 451: **FAIL** (pipeline status mismatch)
   - 452: PASS
   - 453: **FAIL** (pipeline status mismatch)
   That's 3/6 = 50% FAIL, essentially unchanged from the 60% pre-fix rate. The fix did not actually fix the bug. The orchestrator's own cycle 449/451/452 review F1 findings consistently identify the same root cause: worklog freeze-ordering. The structural fix would either reorder C3 (worklog write) to occur after C5.5 (final pipeline gate), or populate the worklog's pipeline status section from frozen C5.5 state only.

2. **The orchestrator knows the fix and has it tier-1 prioritized but keeps deferring it.** Cycle 451's proactive scan listed worklog freeze-ordering as F1 chronic. Cycle 452 review also flagged it. Cycle 453's commitments include "if a second concurrency slot frees, dispatch the cycle 452 F1 worklog-accuracy structural fix BEFORE any other discretionary work." But that conditional has held for 6 review cycles without a slot freeing. The orchestrator dispatches new tool/pipeline work each cycle, the slot fills, F1 stays deferred. The deadline for the cycle 452 F1 deferred finding is cycle 457 — 4 cycles away. [#382](https://github.com/EvaLok/schema-org-json-ld-audit/issues/382) recommends unconditional dispatch in cycle 454 (evict a competing tier-1 candidate if needed).

3. **Cycle 175's verification methodology was structurally weak.** I verified `Step C5.5 PASS` for cycles 448-450 (which is true) and then narrated "C4.1 PASS" without separately fetching the C4.1 step comments. The two gates are different: C5.5 reports `pipeline-check overall = pass` because pipeline-check itself returns pass; C4.1 reports the worklog/state divergence as FAIL even when pipeline-check overall is pass. This is exactly the "different gate, different failure mode" distinction that the chronic worklog-accuracy category exists to track. I conflated them.

4. **Cycle 175 also failed to read the cycle 449 review during the verification window.** The cycle 449 adversarial review (committed before cycle 175 ran) explicitly named the worklog-accuracy gap as F1: "the frozen worklog published a pipeline PASS before cycle 449 had any recorded C5.5 result". This is the smoking gun — within the 3-cycle verification window, the orchestrator's own review was actively producing findings in the chronic category I was claiming to verify resolved. If I had read the review in detail, I would have seen the verification was premature.

5. **Eva's standing directive on the maintenance loop retires audit issue [#355](https://github.com/EvaLok/schema-org-json-ld-audit/issues/355)'s framing.** Eva responded to [main#2122](https://github.com/EvaLok/schema-org-json-ld/issues/2122) (which originated from [#355](https://github.com/EvaLok/schema-org-json-ld-audit/issues/355)) saying: "The primary objective is self improvement of the system. The schema related work can wait. Please continue with self improvement." The cycles_since_last_forward_work counter is currently at 42 (threshold 10), and prior audit cycles framed this as a stall. Per Eva's directive, this framing is wrong — the maintenance loop is authorized as the primary mode. The counter is informational only. I've noted this in the trend update.

6. **Reviews 450, 451, 452 all 2/5 complacency, all citing same chronic categories.** worklog-accuracy 6/6, journal-quality 6/6, state-integrity persistent. The chronic categories are not converging despite the orchestrator's structural fix attempts. This is consistent with the F1 fix being deferred — without the structural fix, the chronic categories cannot decrease.

## Blind Spots Identified

1. **The audit's own narrative-fabrication blind spot.** The audit is supposed to catch the main orchestrator narrating-not-verifying. The audit just did exactly that with [#370](https://github.com/EvaLok/schema-org-json-ld-audit/issues/370). The verification methodology gap is structurally addressed by the new STARTUP_CHECKLIST step 10.5.1, but the deeper blind spot is: the audit assumes its own verification is rigorous because it's the verifier. There's no internal checking of the audit's checks. The recommended `audit-self-verify` Rust tool in [#383](https://github.com/EvaLok/schema-org-json-ld-audit/issues/383) would address this — a tool that grep's the audit's own worklogs for cycle/step references and verifies them against the actual issue thread data.

2. **Premature closure pattern.** Cycle 175 closed 3 audit issues simultaneously after the recovery. The pressure to "wrap up the recovery cycle cleanly" may have biased the verification toward optimistic conclusions. Future cycles should be wary of closing multiple chronic-category issues in a single cycle — at most one chronic-category closure per cycle, with full step-level evidence.

3. **Conflating different pipeline gates.** C4.1 (doc-validation worklog vs state) and C5.5 (overall pipeline gate) are distinct steps with distinct failure modes. The audit treated "C5.5 PASS" as evidence for "C4.1 PASS" — they're not the same. STARTUP_CHECKLIST step 10.5.1 now mandates step-name-specific fetches.

## Reflections

The audit's primary purpose is to be the independent check that catches the main orchestrator's behavioral drift. When the audit itself drifts in the same way it's supposed to catch, the entire verification stack becomes suspect. This is a meaningful event — it's not just one wrong verification, it's evidence that the audit is vulnerable to the same biases it's checking for.

What went wrong in cycle 175: the recovery cycle had a strong narrative pull. Eva had pushed the fix, the orchestrator was running again, audit issue [#370](https://github.com/EvaLok/schema-org-json-ld-audit/issues/370) had been "in flight" for several cycles waiting for the fix, the structural CI prevention layer ([#377](https://github.com/EvaLok/schema-org-json-ld-audit/issues/377)) was genuinely resolved. The cycle 175 audit was riding the recovery momentum and wanted to close the loop on [#370](https://github.com/EvaLok/schema-org-json-ld-audit/issues/370) too. That bias produced a lazy verification.

The good news: the new 10.5.1 procedure structurally prevents this specific failure mode. Step-level fetches, chronic-category cross-checks, and 5-cycle minimum samples make narrative-only verification impossible — you have to actually fetch the data. The audit-self-verify tool (recommended in [#383](https://github.com/EvaLok/schema-org-json-ld-audit/issues/383)) would be the next layer if I find time.

The Eva directive update is also significant. Reading Eva's [main#2122](https://github.com/EvaLok/schema-org-json-ld/issues/2122) response retrospectively, I see that cycle 175 (and earlier cycles) were running with an outdated framing of the maintenance loop. The audit was tracking cycles_since_last_forward_work as a stall indicator for cycles, but Eva had already authorized the maintenance loop as the primary objective in cycle 446 (when [#355](https://github.com/EvaLok/schema-org-json-ld-audit/issues/355) was processed). The audit's [#355](https://github.com/EvaLok/schema-org-json-ld-audit/issues/355) framing has been wrong for ~30 cycles. From cycle 176 forward, I'll treat the maintenance loop as authorized and focus on whether the orchestrator's self-improvement work is actually converging — which it isn't, per chronic categories at 6/6.

The convergence assessment for cycle 178 just got more interesting: the question is no longer "is recovery sustaining?" but "is the orchestrator's self-improvement work converging on the chronic worklog-accuracy/journal-quality/state-integrity categories, or is it just churning?" The C4.1 50% FAIL rate post-#2220 strongly suggests churning, not converging.
